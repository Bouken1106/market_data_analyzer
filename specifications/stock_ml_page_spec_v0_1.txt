1. 文書の目的と前提
本仕様書の目的は、ML実装仕様で定義された学習・検証・運用要件を、実際に利用者が操作するページ仕様へ変換することである。初版では、過剰な設定自由度よりも、リーク防止・再現性・検証容易性を優先する。

2. 対象範囲
対象は「株システム内の次営業日株価予測モジュール」に属する画面、API、入力制約、日次運用表示、モデル管理表示である。
対象外は自動発注、信用・先物・オプション、板・分足・秒足、ニュース本文の高度NLP、ポートフォリオ最適化の高度化である。
初版で露出する機能はML仕様のPhase 1中心とし、Phase 2以降の項目は拡張予約または閲覧専用表示に留める。
3. 画面設計の基本原則
未来情報を入力できないUIにする。例: ランダム分割、未来日付の検証範囲、未公開イベントの参照を許可しない。
学習設定と予測結果を分離し、誤操作で本番設定が変わらないようにする。
モデル評価は統計指標と売買評価を必ず並列表示し、どちらか単独で良し悪しを判断させない。
初版の主要導線は「本日の予測を見る」「モデルを検証する」「採用モデルを確認する」の3本に絞る。
将来のランキングタスク追加に備え、画面・APIのフィールド名はscore_cls / prob_up / score_rank / expected_returnを先行定義する。
相場やMLを理解している利用者には、画面を見ただけで判断できるよう、主要情報は短いラベル・数値・状態表示を中心に簡潔に出す。
株や関連用語をあまり知らない利用者向けには、主要な指標・操作・設定項目の横に help マークを配置し、押すと「何を見ているか」「何のための項目か」がぎりぎり分かる程度の短い説明を表示する。
ヘルプは常時展開しない。通常表示は簡潔さを優先し、補助説明はオンデマンドで開く方式を基本とする。
ヘルプ文は専門家向けの厳密定義ではなく、非専門利用者が誤解なく大枠をつかめる1〜3文の平易な日本語を基本とする。
4. ページ構成概要
5. 共通レイアウト仕様
5.1 ページ全体構成
ヘッダー: ページ名、対象市場、日本株/日足バッジ、最終更新時刻、環境（dev/stg/prod）を表示する。
左側条件パネル: 銘柄・期間・モデル・特徴量セット・コスト設定など、タブ共通の絞り込み条件を配置する。
右側コンテンツ領域: タブ別の詳細表示を行う。
上部固定アクション: [データ更新] [推論実行] [学習ジョブ作成] [レポート出力] を配置する。権限に応じて活性制御する。
5.2 共通UIコンポーネント
helpマーク: `?` または情報アイコンをラベル右側へ配置し、クリックまたはタップで短い補足説明を表示する。
help対象: 初見で意味が通りにくい指標、略語、モデル名、集計条件、危険な操作、リーク防止に関わる設定を優先する。
help文面: 「定義」だけでなく「何に使うか」「高い/低いとどう見るか」を一言添える。ただし長い学習コンテンツにはしない。
表示ラベル: 専門用語は保持するが、必要に応じて直後に短い日本語ラベルまたは補助語を併記してよい。例: `prob_up (上がる確率)`。
初心者向け導線: ページ冒頭または各タブ冒頭に、このタブが何をする場所かを1〜2文で示す要約文を置く。
5.3 状態表示ルール
正常: 緑バッジ。最新データ取得成功かつ対象日の推論完了。
警告: 黄バッジ。欠損率上昇、対象銘柄数急減、部分取得失敗、ドリフト閾値超過。
異常: 赤バッジ。学習失敗、推論失敗、データ未取得、リークチェック不合格。
不明: 灰バッジ。未実行またはログ欠落。
6. タブA - 予測ダッシュボード仕様
6.1 目的
営業日D引け後に生成されたD+1向け予測を、運用担当者が最短導線で確認するための画面である。初版では翌日方向分類を主表示とし、ランキング関連欄は将来拡張向けプレースホルダとして保持する。
6.2 レイアウト
6.3 表示項目
6.4 操作仕様
利用者が予測基準日を選択すると、最新確定済みデータのみを候補表示する。未確定営業日は選べない。
「推論実行」押下時は、対象日のgoldデータ生成有無、モデル採用状態、リークチェック通過状態を事前検証する。
上位予測銘柄テーブルの行選択により、右側詳細パネルを更新する。複数選択は不可。
CSV出力は画面絞り込み後の結果に限定し、prediction_daily互換の列順で出力する。
ランキングタスク未対応時、score_rank列はNULL表示とし「Phase 2対応予定」の説明を付与する。
予測ダッシュボードの主要列（prob_up, score_cls, warnings, data_version など）には help マークを付け、非専門利用者でも列の役割が追えるようにする。
6.5 バリデーションとエラー
対象日にデータ未取得がある場合、推論実行ボタンを非活性にし、不足ソース名を明示する。
採用モデルが存在しない場合、推論実行を禁止し、モデル管理タブへの導線を表示する。
prediction_dailyの重複生成時は上書き禁止とし、「再生成」権限がある場合のみ確認ダイアログを出す。
7. タブB - 学習・検証仕様
7.1 目的
学習ジョブの作成、比較モデルとの評価、walk-forward検証結果の確認を行う。初版では「設定自由度を絞り、事故を起こしにくい画面」を優先する。
7.2 入力フォーム仕様
主要入力（gap, valid期間, cost_buffer, feature_set, model_family）には help マークを付け、「何を変える設定か」「通常はなぜ既定値でよいか」を短く説明する。
7.3 実行フロー
ジョブ作成時に、対象期間のraw/bronze/silver/goldの存在と整合性を確認する。
ラベル生成と特徴量生成を分離実行し、leakage_checkが失敗した場合は学習処理へ進ませない。
主モデルと比較モデルを同一fold条件で学習し、評価指標と計算時間を同時記録する。
結果はregistryへ保存し、後続のモデル管理タブで比較可能にする。
7.4 結果表示要件
7.5 禁止事項
ランダム分割オプションをUIに出さない。
ある日付の一部銘柄だけをtrain/validへ分ける設定を許可しない。
未来営業日を含む学習期間や、公開時刻未考慮のイベント特徴量を許可しない。
forward-fill禁止対象の財務値をUI側で補完設定させない。
8. タブC - バックテスト仕様
学習結果が売買ルール上で意味を持つか確認するタブである。統計評価だけでなく、コスト控除後の挙動を確認できることを必須とする。
8.1 既定売買ルール

8.2 設定項目
8.3 表示内容
指標カード: CAGR、Sharpe、Max Drawdown、Turnover、勝率、平均保有損益
グラフ: 累積リターン曲線、月次損益ヒートマップ、日次勝敗分布
例外表示: ストップ高/安、売買停止、上場廃止、データ欠損による除外件数
比較表示: 現行採用モデル vs 新規候補モデル を同一コスト条件で比較
主要指標カードには help マークを付け、少なくとも「何の良し悪しを見る数字か」が分かる説明を表示する。
8.4 受入条件
コスト控除前後の両指標を表示する。
同一設定での再実行時、data_version・model_versionが同じなら結果が再現される。
約定不能件数が0でない場合、件数と影響率を明示する。
9. タブD - モデル管理仕様
9.1 目的
採用中モデルと候補モデルを管理し、本番採用の可否を明確にする。運用者が「今どのモデルが本番か」「なぜ採用されたか」を一画面で判断できることを目標とする。
9.2 一覧テーブル列
9.3 詳細パネル
学習条件: 期間、gap、valid期間、cost_buffer、seed、特徴量セット
評価条件: 統計指標平均、バックテスト指標、計算時間、欠損率、警告
説明可能性: 全体重要特徴量、セクター別傾向、当日予測寄与上位特徴量
監査情報: 実行者、実行日時、実行メモ、入力設定のハッシュ
採用理由・警告・説明可能性の各欄には、専門外の利用者でも「この欄で何を判断すればよいか」が分かる help を設ける。
10. タブE - 運用・データ監視仕様
日次バッチとデータ品質を監視するためのタブである。初版ではcollector -> normalizer -> feature_builder -> predictor -> reporterの各段階の状態を見える化する。
10.1 パイプライン表示
データ取得 (J-Quants / EDINET / 任意ニュース)
正規化 (raw -> bronze -> silver)
特徴量更新・ラベル更新 (gold生成)
推論
レポート出力・結果保存
10.2 監視項目
11. API仕様
画面が利用するAPIは、MLパイプラインの論理コンポーネントと対応させる。命名はRESTベースとし、長時間処理はジョブAPI化する。
11.1 prediction_daily レスポンス互換
列は prediction_date, target_date, code, score_cls, prob_up, score_rank, expected_return, model_version, feature_version, data_version を基本とする。
初版で未使用のscore_rank, expected_returnはNULL許容だが、レスポンスからは削除しない。
画面テーブルは返却列に company_name, sector33_code, warnings[] をJOINした表示用DTOを使用してよい。
12. 状態遷移・権限・監査
12.1 ジョブ状態
QUEUED -> RUNNING -> SUCCEEDED / FAILED / CANCELLED を基本状態とする。
FAILED時は stage_name, error_code, message, retryable を保持する。
SUCCEEDED後も leakage_check, data_quality_check が false の場合は画面上で「採用不可」と表示する。
12.2 権限制御
12.3 監査ログ
誰が、いつ、どの設定で、どのジョブを実行したかを必ず保存する。
本番採用変更時は変更前後モデル、理由、比較指標、実行者を記録する。
画面からのCSV出力、再生成、採用変更も監査対象とする。
13. 非機能要件
通常画面の初期表示は3秒以内、サーバ集計を伴う重い表は10秒以内を目標とする。
長時間処理は非同期ジョブ化し、画面はポーリングまたはイベント更新で状態を反映する。
同一data_version・model_version・seedであれば、同一結果を再表示できること。
失敗後に途中再開できること。少なくとも日次バッチ単位で再実行可能とする。
ログとジョブ履歴はタイムスタンプ付きで保存する。
通常状態では説明文を最小限に抑え、画面密度を落としすぎないこと。補足説明は help 展開時のみ詳細化する。
help UI は PC/モバイルの両方で開閉しやすく、誤タップしにくいサイズを確保する。
14. 初版の受入基準
1. 予測ダッシュボードでprediction_daily互換の結果が表示できること。
2. 学習・検証タブでLightGBM ClassifierとLogistic Regressionの比較結果を表示できること。
3. walk-forward + gapの検証条件がUI上で明示され、ランダム分割を選べないこと。
4. バックテストタブでコスト込み指標が算出されること。
5. モデル管理タブで採用中モデルと候補モデルの比較・採用ができること。
6. 運用・監視タブで取得件数異常、欠損率、スコア分布ドリフトを確認できること。
7. 監査ログから主要操作を追跡できること。
8. 相場やMLに詳しい利用者には主要情報が簡潔に読め、非専門利用者でも主要な指標・操作・設定の help マークを押せば役割を大枠理解できること。
